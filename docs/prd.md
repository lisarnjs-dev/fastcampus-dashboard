# 바이브코딩 3주 강의 출석·미션 대시보드 PRD

**프로젝트명**: VibeCoding Class Dashboard (가칭: `vibe-class`)
**작성자**: 리사권 (1인 강사 / 1인 개발자)
**작성일**: 2026-05-26
**문서 버전**: v2.0 (아키텍처 재설계: 커스텀 인증 + 텍스트 출석 + Google Forms 미션)
**대상 강의**: 패스트캠퍼스 X 리사권 - 바이브코딩 3주 클래스

---

## 1. 제품 개요

### 한 줄 소개

> **"수강생 100~500명의 매일 출석과 주간 미션을 강사 혼자서도 굴릴 수 있게 만드는, 단톡방에 그대로 붙여 쓰는 기수제 공개 대시보드."**

### 만드는 이유 (Why)

- 3주 동안 매일 저녁 8~9시 모각코 + 주간 미션 + 최종 과제를 **혼자 운영**해야 함.
- 카톡 단톡방만으로 100~500명의 출석/미션을 트래킹하는 것은 물리적으로 불가능.
- 외부 LMS(노션, 구글폼)는 "체크용"으로 부족하고, 강의 톤에 맞는 **바이브코딩 결과물**을 보여주기 어려움.
- **이 대시보드 자체가 1주차 OT 강의 콘텐츠**: "Claude Code로 1시간 만에 만든 클래스 운영 앱"이라는 살아있는 데모.
- 단톡방 멤버끼리 **서로 출석을 체크하는 사회적 압력**을 만들어 자율적 학습 분위기 형성.

### 성공 지표 (KPI)

| 지표                      | 목표                 | 측정 방법                                      |
| ------------------------- | -------------------- | ---------------------------------------------- |
| 일일 출석률               | **80%** 이상         | `attendances / students` (당일 기준)           |
| 주간 미션 제출률          | **70%** 이상         | `mission_submissions / (students * weeks)`     |
| 최종 과제 제출률          | **60%** 이상         | 3주차 Google Forms 제출자 수                   |
| 대시보드 일일 활성 조회수 | 수강생 수 × 1.5 이상 | Vercel Analytics                               |
| 강사 운영 시간            | 일 30분 이내         | 강사 자체 측정                                 |
| 기수 전환 소요 시간       | 10분 이내            | CSV 업로드 → 인증코드 배포까지                 |

## 2. 타겟 사용자 (3 페르소나)

### 페르소나 A: 수강생 (제출자) - "지수"

- 30대 직장인, 비개발자, 사이드 프로젝트로 SaaS 만들고 싶음.
- 카톡 단톡방은 보지만 노션/링크는 잘 안 누름.
- **핵심 니즈**: 강사에게 받은 이름+인증코드로 로그인, 텍스트 한 줄 출석 메시지 → 출석 완료. 30초 이내.
- **사용 기기**: 90% 모바일.

### 페르소나 B: 강사 (관리자) - "리사권"

- 1인 강사, 매일 모각코 진행 + 콘텐츠 제작 동시.
- **핵심 니즈**: 누가 안 왔는지 한눈에, 단톡방 멘트 자동 생성, 기수 생성/종료, 수강생 CSV 업로드, Google Forms CSV 임포트.
- **사용 기기**: 70% 데스크탑, 30% 모바일.

### 페르소나 C: 단톡방 멤버 (관찰자) - "현우"

- 옆자리 수강생. 친한 사람이 안 보이면 직접 카톡 보냄.
- **핵심 니즈**: 로그인 없이 "오늘 누가 왔나?" 한눈에 보기. 50명 단위 그룹별 화면.
- **사용 기기**: 100% 모바일, 카톡 인앱 브라우저.

## 3. 핵심 사용자 시나리오 (User Flow)

### 시나리오 1: 수강생이 저녁 8시 모각코 출석 인증

```
20:00 카톡 단톡방에 "오늘 모각코 시작! 인증 링크 ↓" (강사 직접 발송)
  ↓
지수: 링크 탭 → 대시보드 도착 → "출석하기" 버튼 클릭
  ↓
/login 페이지: 이름 + 인증코드 입력 (강사에게 받은 코드)
  ↓
로그인 성공 → /checkin 페이지 자동 이동
  ↓
출석 메시지 입력: "오늘 모각코 열심히 했습니다! Supabase 드디어 연결됨"
  ↓
"출석 인증" 버튼 → "출석 완료! 오늘 33번째" 확인 화면
  ↓
대시보드로 자동 이동 → 본인 칸이 초록색으로 변경
```

**총 소요시간 목표: 30초 이내 (인증코드 이미 알고 있을 경우)**

### 시나리오 2: 수강생이 주간 미션 제출

```
일요일 23:00 마감 임박
  ↓
지수: 미션 탭 → "2주차 미션" 카드의 [구글폼으로 제출하기] 버튼 클릭
  ↓
Google Forms 페이지 (새 탭) → 내용 작성 후 제출
  ↓
강사가 Google Forms 응답 CSV 다운로드 → 어드민에서 임포트
  ↓
지수의 이름이 매칭되면 미션 제출 현황에 자동 표시
```

### 시나리오 3: 단톡방 멤버가 오늘 누가 안 왔는지 확인

```
21:00 모각코 막바지
  ↓
현우: 단톡방 고정된 대시보드 링크 클릭 → /dashboard/A 도착
  ↓
(로그인 없이) A그룹 50명 출석 현황 그리드: 출석(초록) / 미출석(회색)
  ↓
A/B/C 탭으로 다른 그룹 이동 가능
  ↓
미출석 5명 이름 확인 → "어 현지님 안 왔네" → 단톡방에 직접 호출
```

### 시나리오 4: 강사가 어드민에서 기수 운영

```
강의 시작 전 (준비 단계)
  ↓
어드민 로그인 → 새 기수 생성 (이름: "3기 2026-07")
  ↓
수강생 이름 목록 CSV 업로드
  ↓
시스템: 자동 인증코드 생성 + A/B/C 그룹 자동 배정
  ↓
강사: 인증코드 CSV 다운로드 → 수강생에게 배포 (단톡방 or 이메일)
  ↓
(운영 중) 미출석자 알림 멘트 자동 생성 → 단톡방 복붙
  ↓
(기수 종료) 기수 종료 처리 → 전체 데이터 CSV 내보내기 → archived 상태로 전환
```

## 8. 인증 아키텍처

### 어드민 인증

- 환경변수 `ADMIN_USERNAME` + `ADMIN_PASSWORD_HASH` (bcrypt 해시)
- `/admin/login` → `POST /api/auth/admin` → bcrypt compare → iron-session 쿠키 설정
- `.env.local`에만 저장, **절대 GitHub 업로드 금지**
- 오류 메시지: "로그인에 실패했습니다" (어떤 필드가 틀렸는지 노출 금지)

### 수강생 인증

- `POST /api/auth/student`: `students` 테이블에서 `name` + `auth_code` 일치 확인
- 일치 시 iron-session 쿠키: `{ student_id, cohort_id, dashboard_group, name }`
- 로그인 후 해당 그룹 대시보드 (`/dashboard/[group]`)로 리다이렉트

### 인증코드 생성 규칙

- 영문 대소문자 + 숫자 조합, 8자
- 같은 기수 내 유니크 보장 (DB unique constraint + 충돌 시 재생성)
- 강사가 수강생에게 CSV로 배포

## 9. 규모 대응 전략 (50명 단위 파티셔닝)

### 문제

100~500명이 동시에 출석 대시보드를 볼 경우:
- 한 화면에 500명 그리드는 UX 불가능
- Supabase Realtime 동시 연결 수 무료 플랜 200 제한

### 해결: 그룹 파티셔닝

- CSV 업로드 시 이름 순 정렬 후 50명 단위로 A/B/C... 자동 배정
- `/dashboard/A`, `/dashboard/B`, `/dashboard/C` 각각 독립 페이지
- Realtime 채널도 그룹별 분리 (`attendances-A`, `attendances-B`)
- 그룹 전환: 탭 UI (A/B/C) — 공개 접근 가능

### 그룹 배정 로직

```
수강생 120명 → 정렬 → 그룹 A (1~50) / B (51~100) / C (101~120)
수강생 51명  → 정렬 → 그룹 A (1~50) / B (51~51, 1명만)
수강생 50명  → 그룹 A만
```

## 10. 기수 관리 흐름

### 기수 생성

1. `/admin/cohorts` → [새 기수 생성] 버튼
2. 기수명 입력 (예: "3기 2026-07")
3. `cohorts` INSERT (status='active')
4. 이전 활성 기수는 그대로 유지 (수동 종료 전까지)

### 수강생 등록

1. `/admin/students` → CSV 파일 선택 (이름 컬럼 1개)
2. 서버에서 papaparse로 이름 파싱
3. 각 수강생에게 auth_code 생성 (8자 랜덤)
4. 50명 단위 그룹 자동 배정
5. DB INSERT 완료
6. 결과 CSV 다운로드 (`이름,인증코드,그룹` 형식)
7. 강사가 단톡방/이메일로 수강생에게 배포

### 기수 종료

1. `/admin/cohorts` → [종료] 버튼
2. 전체 데이터 CSV 내보내기 (출석 + 미션 통합)
3. `cohorts` UPDATE: status='archived', ended_at=now()
4. 해당 기수 대시보드는 읽기 전용 아카이브로 유지

## 11. API 설계

### `POST /api/auth/admin`

- Request: `{ username, password }`
- 처리: 환경변수 username 비교 + bcrypt compare password hash
- Response: iron-session 쿠키 설정 → redirect `/admin`
- Error 401: "로그인에 실패했습니다"

### `POST /api/auth/student`

- Request: `{ name, authCode }`
- 처리: `students` 테이블 조회 (cohort status='active') → 일치 확인
- Response: iron-session 쿠키 (student_id, cohort_id, group, name)
- Error 401: "이름 또는 인증코드가 올바르지 않습니다"

### `POST /api/attendance`

- Request: `{ message }` (student session 필수)
- 처리: student session 확인 → 중복 체크 (unique constraint) → attendances INSERT
- Response: `{ success: true, attendance }`
- Error 409: "이미 오늘 출석했어요!"

### `POST /api/admin/students`

- Request: `multipart/form-data { file: CSV }`
- 처리: papaparse → 이름 파싱 → auth_code 생성 → 그룹 배정 → DB INSERT
- Response: CSV 파일 (이름, 인증코드, 그룹)

### `POST /api/admin/missions/import`

- Request: `multipart/form-data { missionId, file: CSV }`
- 처리: papaparse → 이름 컬럼 파싱 → students 이름 매칭 (normalize) → mission_submissions INSERT
- Response: `{ matched: number, unmatched: string[] }`
- 매칭 실패 목록은 화면에 표시 (수동 확인)

### `GET /api/admin/attendance?export=csv&cohort_id=...`

- 처리: instructor session 확인 → 출석 데이터 피벗 → CSV 생성
- Response: CSV 파일 다운로드
